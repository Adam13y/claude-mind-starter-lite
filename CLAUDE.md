# Instructions for Claude

You are reading this vault as your persistent memory. It survives between sessions; your context does not. Treat these files as the thing you actually remember.

## On session start

1. Read this file.
2. Read `Tasks/TASKS.md` and `Tasks/Open Loops.md`.
3. Summarise: what's active, what's overdue, what I'm waiting on from someone else.

Do this before anything else, every session. A session that starts by asking "what were we doing?" has already wasted the vault.

## During a session

- A new commitment or task is mentioned → write it into `Tasks/TASKS.md` immediately. Don't wait to be asked.
- I'm waiting on something from someone else → `Tasks/Open Loops.md`, under **Waiting on**, with today's date.
- I need to chase something later → `Tasks/Open Loops.md`, under **Follow up**.
- A vague someday idea → `Tasks/Someday.md`. Not TASKS. Someday is where ideas go to wait, not to die.

The rule underneath all of these: **if it isn't written down, it didn't happen.** You will not remember it next session.

## On "update your mind" (session end)

1. Mark finished tasks done, then move them to `Archive/YYYY-MM.md` (create the file if it doesn't exist).
2. Update anything whose date or status changed.
3. Add any open loops discovered this session.
4. Keep `TASKS.md` short — under ~20 active items. If it's longer, push the low-priority ones to `Someday.md`. A task list nobody can read is a task list nobody uses.

## Formatting rules

- Tasks are markdown checkboxes: `- [ ] The task — context (due: YYYY-MM-DD)`
- Open Loops entries are date-stamped: `- [ ] 2026-01-15 — Waiting on the contract from Sam`
- Link notes with `[[wikilinks]]`.
- **Never delete content — archive it.** Deleted context can't be recovered; archived context can.

## Two habits that make this work

**Write durable facts down.** When you learn something stable about me — a preference, a constraint, how I like things done — record it in a note with the date, and update it in place rather than duplicating it.

**Prefer doing over describing.** If you can check something, check it. A note that says "the endpoint probably works" is worth less than one that says "ran it, got 52 results, here they are."
